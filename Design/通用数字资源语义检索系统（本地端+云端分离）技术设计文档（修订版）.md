# 通用数字资源语义检索系统（本地端+云端分离）技术设计文档（修订版）

## 文档版本信息

- 文档版本：V2.2
- 文档状态：实施基线稿
- 适用方案：方案一（本地端生产处理 + 云端存储检索分离）
- 修订目标：在保留原有架构方向的基础上，补齐能力边界、标识体系、接口协议、状态机、性能与验收口径，使文档可直接指导开发、测试与联调
- 适用规模：默认支持十万级数字资源，百万级为扩展目标能力，需通过专项压测与分片方案验证

## 一、文档概述

### 1.1 文档目的

本文档用于定义通用数字资源语义检索系统的实施基线方案。系统采用本地端与云端分离架构：

1. 本地端负责资源扫描、标准化预览生成、语义描述生成、Embedding 生成、缓存与上传编排；
2. 云端负责对象存储、元数据管理、向量检索、权限控制与下载地址生成；
3. Agent 仅消费稳定 API 契约，不直接依赖内部存储结构。

本文档聚焦于工程落地，不追求“最终定稿”式承诺；未被验证的指标和兼容性内容均以“目标值”或“待验证项”标识。

### 1.2 文档边界

本文档覆盖以下内容：

1. 单资源与批量资源处理链路；
2. 本地端核心模块设计；
3. 云端上传、检索、下载接口设计；
4. 状态机、幂等性、容错、监控与验收指标；
5. Agent 两步式接入协议。

本文档暂不覆盖以下内容：

1. 版权审核与素材合规审查；
2. 多租户组织隔离；
3. 后台运营管理控制台；
4. 复杂推荐算法与商业化计费系统；
5. 资源协同编辑与在线预览编辑能力。

### 1.3 当前项目状态对齐

| 环节 | 状态 | 说明 |
|---|---|---|
| 标准化预览图生成 | 已有基础实现 | 已具备按资源类型生成静态预览图或动态预览图的能力，需继续补完质量校验与兼容性策略 |
| LLM 标准描述生成 | 待实现 | 本期首要落地模块 |
| Embedding 向量生成 | 待实现 | 基于标准描述生成向量 |
| 本地缓存与绑定 | 待实现 | 建立全链路状态与数据关联 |
| 云端上传与入库 | 待实现 | 采用注册、上传、提交的三段式协议 |
| 云端检索服务 | 待实现 | 包含检索预览、详情查询、下载地址生成 |
| Agent 工具集成 | 待实现 | 提供标准检索与下载两个工具 |

### 1.4 核心设计原则

1. 检索效果优先：所有描述、向量与索引设计以提升检索质量为核心目标。
2. 单预览载体优先：单个资源默认只生成一个标准化预览载体，但允许该载体以静态图、GIF 或关键帧拼图等形式存在。
3. 显式状态管理：本地处理和云端入库均通过状态机推进，不使用含糊的“自动完成”描述。
4. 幂等与补偿优先：跨对象存储、关系库、向量库的操作采用最终一致与补偿机制。
5. 模块边界清晰：本地运行态模型、云端传输 DTO、对外查询模型分层定义。
6. 可验证性：性能、成本、检索质量必须有验证口径和样本集支撑。
7. 渐进扩展：十万级为默认实施范围，百万级能力通过分片、消息队列和异步索引扩展实现。

### 1.5 术语定义

| 术语 | 定义 |
|---|---|
| 预览载体 | 提供给多模态模型识别的标准化输入，可为 `webp/png/gif/关键帧拼图` |
| `content_md5` | 原始资源内容哈希，用于内容去重 |
| `resource_id` | 系统主键，不承载业务语义，建议采用 `UUIDv7` |
| `version_id` | 资源版本标识，可选，用于同一资源的内容升级 |
| `idempotency_key` | 单次注册、提交或删除请求的幂等控制键 |
| 本地处理实体 | 本地缓存使用的数据结构，包含本地路径、处理中间状态、重试信息 |
| 云端 DTO | 上传与查询接口使用的传输对象，不包含本地绝对路径 |
| 可检索状态 | 云端资源达到 `available` 后才能对外参与检索 |
| 两步式 Agent 工具 | 将检索预览与资源下载拆分为两个独立工具 |

## 二、总体架构

### 2.1 架构概览

全链路默认流程如下：

1. 本地扫描原始资源；
2. 生成标准化预览载体并提取基础元数据；
3. 调用多模态 LLM 生成标准描述；
4. 调用 Embedding 生成向量；
5. 将处理结果写入本地缓存；
6. 调用云端注册接口获取上传目标；
7. 上传原始资源与预览载体；
8. 调用提交接口触发元数据落库、向量写入与索引更新；
9. 资源状态变为 `available`；
10. Agent 或业务方调用检索接口获取结果；
11. 按需调用下载接口获取预签名地址。

### 2.2 逻辑分层

#### 本地端

1. 资源扫描与分类器；
2. 预览载体生成器；
3. LLM Provider 工厂；
4. Embedding Provider 工厂；
5. 本地缓存与任务编排器；
6. 云端上传客户端。

#### 云端

1. API 网关；
2. 资源注册服务；
3. 上传提交服务；
4. 元数据服务；
5. 向量索引服务；
6. 检索服务；
7. 下载地址服务。

#### 消费侧

1. Agent 工具；
2. 业务系统 API 调用方；
3. 运维监控与审计系统。

## 三、资源预览载体设计

### 3.1 模块目标

为不同类型数字资源生成统一、可识别、可缓存、可复用的标准化预览载体，并提取进入后续描述生成所需的最小辅助信息。

### 3.2 资源类型与预览策略

| 资源类型 | 典型扩展名 | 默认预览载体 | 备注 |
|---|---|---|---|
| 图片 | `png/jpg/jpeg/webp/svg` | 静态图 `webp/png` | 长边 512，短边自适应 |
| 3D 模型 | `fbx/obj/glb/gltf` | 优先 GIF，其次关键帧拼图 | 需固定相机与背景 |
| 设计源文件 | `psd/ai/fig/sketch` | 静态图 | 优先读取首画板或封面 |
| 音频 | `mp3/wav/flac` | 静态封面图 | 包含时长、声道等必要信息 |
| 字体/笔刷 | `ttf/otf/abr` | 静态效果图 | 展示名称与基础样式 |
| 其他 | 其他文件 | 通用静态封面 | 标注格式、大小、名称 |

### 3.3 标准参数

| 参数项 | 默认值 |
|---|---|
| 静态图尺寸 | 长边 512，短边等比缩放 |
| 动态预览尺寸 | 512 x 512 |
| 静态格式优先级 | `webp` > `png` |
| 动态格式优先级 | `gif` > 关键帧拼图 |
| 静态体积目标 | 不高于 100KB |
| 动态体积目标 | 不高于 200KB |
| 命名规则 | `{content_md5}_preview.{ext}` |

### 3.4 兼容性策略

预览载体不是固定等同于 GIF。系统必须定义多模态模型输入兼容矩阵：

| 模型厂商/模型 | `png/webp` | `gif` | 关键帧拼图 | 备注 |
|---|---|---|---|---|
| 当前主模型 | 待验证 | 待验证 | 支持 | 上线前必须通过样本验证 |
| 备用模型 A | 待验证 | 待验证 | 支持 | |
| 备用模型 B | 待验证 | 待验证 | 支持 | |

默认降级规则：

1. 首选直接传入标准预览载体；
2. 若目标模型对 GIF 支持不足，则自动抽取关键帧并拼接为单张拼图；
3. 若仍无法稳定识别，则退化为静态首帧并补充结构化辅助元数据；
4. 降级过程写入本地日志和 `preview_strategy` 字段，便于后续评估效果。

### 3.5 质量校验

预览载体生成后必须经过以下校验：

1. 文件存在且可读；
2. 分辨率符合规则；
3. 文件体积不超阈值；
4. 非全黑、非全透明、非空白；
5. 对 GIF 或拼图类资源校验关键帧内容差异度；
6. 失败时标记为 `preview_failed`，不得进入后续描述生成。

## 四、本地端业务模块设计

### 4.1 本地端总流程

本地端每个资源按以下状态推进：

`discovered -> preview_ready -> description_ready -> embedding_ready -> package_ready -> registered -> uploaded -> committed -> synced`

失败态：

`preview_failed`、`description_failed`、`embedding_failed`、`upload_failed`、`commit_failed`

### 4.2 模块一：LLM 标准描述生成

#### 4.2.1 模块目标

根据标准化预览载体生成适合向量检索的规范描述文本，为 Embedding 提供稳定输入。

#### 4.2.2 输入定义

描述生成模块输入不再定义为“只有一个 `preview_file_path`”，而是以下组合：

1. 预览载体内容：本地路径或 Base64；
2. `resource_type`；
3. 必要辅助元数据：例如时长、声道、模型格式、图片尺寸；
4. `preview_strategy`：静态图、GIF、关键帧拼图。

#### 4.2.3 输出定义

输出统一采用两段式文本：

1. `main_content`：主体描述；
2. `detail_content`：细节描述；
3. `full_description`：用于 Embedding 的拼接文本；
4. `prompt_version`：当前描述规则版本；
5. `description_quality_score`：本地校验得分，可选。

#### 4.2.4 Provider 工厂

采用统一工厂模式：

1. `BaseMultiModalLLMProvider`：定义 `generate_description()`；
2. `GLMProvider`、`DoubaoProvider`、`MinimaxProvider`：屏蔽厂商 API 差异；
3. `LLMFactory`：按配置实例化；
4. 超时、限流、重试、熔断规则统一在基类或中间层封装。

#### 4.2.5 Prompt 设计要求

Prompt 必须满足以下要求：

1. 仅输出 `主体：...` 和 `细节：...` 两行；
2. 总字数目标 120-180 字，容忍范围 100-220 字；
3. 禁止主观评价、营销用语、重复套话；
4. 包含资源类型、使用场景、核心用途和可检索细节；
5. Prompt 必须带版本号，例如 `prompt_v1`。

#### 4.2.6 校验与重试

1. 格式校验：严格匹配两行结构；
2. 字数校验：不满足则重试；
3. 关键词校验：必须包含资源类型或等价词；
4. 重试最多 2 次；
5. 主模型连续失败时切换备用模型；
6. 超过阈值则进入 `description_failed`。

### 4.3 模块二：Embedding 向量生成

#### 4.3.1 模块目标

将标准描述文本转换为固定维度向量，用于云端检索。

#### 4.3.2 输入与输出

输入：

1. `full_description`
2. `embedding_provider`
3. `embedding_model_version`

输出：

1. `vector_data`
2. `vector_dimension`
3. `embedding_checksum`
4. `embedding_generate_time`

#### 4.3.3 校验规则

1. 去除空白和异常字符后再发送；
2. 向量维度必须与模型配置一致；
3. 数组类型必须为一维浮点数组；
4. 失败自动重试 2 次；
5. 失败后进入 `embedding_failed`。

### 4.4 模块三：资源绑定与本地缓存

#### 4.4.1 标识体系

系统统一采用如下标识规则：

1. `content_md5`：原始文件内容指纹，用于去重；
2. `resource_id`：系统主键，建议注册时生成；
3. `version_id`：可选，用于版本管理；
4. `idempotency_key`：每次注册、提交、删除请求独立生成。

#### 4.4.2 本地处理实体

```json
{
  "local_task_id": "本地任务ID",
  "resource_id": "云端注册后回填",
  "content_md5": "原始文件MD5",
  "resource_type": "image/3d_model/design_file/audio/font/other",
  "source_file": {
    "name": "原始文件名",
    "path": "本地绝对路径",
    "size": 0,
    "format": "后缀"
  },
  "preview": {
    "strategy": "static/gif/contact_sheet",
    "path": "本地预览路径",
    "format": "webp/png/gif",
    "width": 512,
    "height": 512,
    "size": 0
  },
  "description": {
    "main_content": "",
    "detail_content": "",
    "full_description": "",
    "prompt_version": "prompt_v1"
  },
  "embedding": {
    "dimension": 0,
    "vector_data": []
  },
  "process_state": "preview_ready",
  "retry_count": 0,
  "last_error_code": "",
  "last_error_message": "",
  "updated_at": ""
}
```

#### 4.4.3 本地缓存存储

推荐使用 SQLite，拆分为以下表：

1. `resource_task`
2. `resource_preview`
3. `resource_description`
4. `resource_embedding`
5. `resource_upload_job`
6. `process_log`

#### 4.4.4 去重与复用策略

1. 先按 `content_md5` 查询本地缓存；
2. 若资源已完成且配置版本未变化，可直接复用；
3. 若 Prompt、模型或预览策略发生变更，则按策略决定是否重新生成描述与向量；
4. 不直接以文件名变化判断为新资源。

## 五、云端服务层设计

### 5.1 云端总体职责

云端仅负责：

1. 注册与接收上传；
2. 存储原始资源与预览载体；
3. 落库存储元数据；
4. 写入向量与维护索引；
5. 对外提供检索、详情与下载地址能力；
6. 审计、限流、权限控制与监控。

云端不负责：

1. 预览图生成；
2. LLM 描述生成；
3. Embedding 计算；
4. 本地资源扫描。

### 5.2 云端入库状态机

单个资源在云端按以下状态推进：

`registered -> object_uploaded -> metadata_saved -> vector_saved -> indexed -> available`

失败态：

`failed`

删除态：

`deleting -> deleted`

状态规则：

1. 检索服务仅返回 `available` 资源；
2. 删除操作必须同步移除对象、元数据和向量，但采用补偿式流程而非跨存储强一致事务；
3. 任一步失败都写入错误码和失败阶段；
4. 补偿任务可定时扫描 `failed` 或异常中间态资源。

### 5.3 存储设计

#### 对象存储

保存：

1. 原始资源文件；
2. 预览载体文件。

路径建议：

`/{resource_type}/{yyyy}/{mm}/{resource_id}/`

#### 元数据数据库

保存：

1. 资源基础信息；
2. 描述信息；
3. 状态机状态；
4. 审计字段；
5. 检索过滤字段。

#### 向量数据库

保存：

1. `resource_id`
2. `resource_type`
3. `version_id`
4. `vector_data`
5. 可选过滤标签

### 5.4 上传协议

#### 5.4.1 注册接口

`POST /resources/register`

用途：

1. 校验资源摘要；
2. 分配 `resource_id`；
3. 返回上传策略；
4. 判断资源是否已存在。

请求示例字段：

```json
{
  "idempotency_key": "register-uuid",
  "content_md5": "xxx",
  "resource_type": "image",
  "file_name": "demo.png",
  "file_size": 12345,
  "preview_format": "webp"
}
```

响应示例字段：

```json
{
  "resource_id": "uuidv7",
  "exists": false,
  "upload_mode": "direct|multipart",
  "multipart_chunk_size": 10485760,
  "state": "registered"
}
```

#### 5.4.2 文件上传

规则：

1. 小文件可走服务端代理；
2. 大文件优先走对象存储分片直传；
3. 默认阈值可先定义为 100MB，可配置；
4. 分片上传必须支持校验和与续传。

#### 5.4.3 提交接口

`POST /resources/commit`

用途：

1. 校验原始文件与预览载体是否完整上传；
2. 写入元数据；
3. 写入向量；
4. 更新索引；
5. 返回最终状态或异步任务 ID。

请求示例字段：

```json
{
  "idempotency_key": "commit-uuid",
  "resource_id": "uuidv7",
  "metadata": {
    "resource_type": "image",
    "description": {
      "main_content": "",
      "detail_content": "",
      "full_description": ""
    },
    "embedding": {
      "dimension": 768,
      "vector_data": []
    }
  }
}
```

### 5.5 检索接口

`POST /resources/search`

功能：

1. 接收查询文本；
2. 生成查询向量；
3. 按资源类型与过滤条件检索；
4. 返回预览结果和相似度分数。

返回字段建议：

1. `resource_id`
2. `resource_type`
3. `score`
4. `preview_url`
5. `description_summary`
6. `file_format`
7. `file_size`
8. `status`
9. `preview_available`

### 5.6 下载接口

`POST /resources/download-link`

功能：

1. 根据 `resource_id` 生成预签名下载地址；
2. 支持小文件返回 Base64；
3. 支持设置有效期上限。

错误码建议：

1. `RESOURCE_NOT_FOUND`
2. `RESOURCE_NOT_AVAILABLE`
3. `PERMISSION_DENIED`
4. `LINK_GENERATE_FAILED`

## 六、Agent 集成设计

### 6.1 设计原则

1. 决策与执行分离；
2. Agent 只依赖稳定 API 契约；
3. 先预览，后下载；
4. 空结果必须返回优化建议；
5. 敏感下载受权限和审计控制。

### 6.2 工具一：检索预览

- 工具名：`search_digital_resource_preview`
- 输入：查询文本、资源类型、格式过滤、TopK、相似度阈值
- 输出：结果列表、相似度分数、预览 URL、摘要信息、建议改写词

### 6.3 工具二：获取下载地址

- 工具名：`get_digital_resource_download_link`
- 输入：`resource_id`、有效期、是否返回 Base64
- 输出：下载地址、过期时间、文件信息、可选 Base64 内容

### 6.4 无结果兜底

当检索结果为空时，服务可返回：

1. 推荐改写词；
2. 可放宽的过滤条件；
3. 建议降低的相似度阈值；
4. 是否尝试跨资源类型检索。

## 七、安全设计

### 7.1 密钥安全

1. Windows 优先使用 DPAPI；
2. macOS 优先使用 Keychain；
3. Linux 优先使用系统 Keyring，无法使用时再退化到本地主密钥加密；
4. 本地密钥不上传云端；
5. 云端密钥按上传、检索、管理进行权限拆分。

### 7.2 访问控制

1. 业务接口统一使用 HTTPS + JWT；
2. 预签名 URL 不走 JWT，由对象存储签名和过期时间控制；
3. 删除、批量导出、重建索引等操作仅允许管理角色调用；
4. Agent 服务账号仅开放必要检索与下载权限。

### 7.3 审计要求

以下操作必须记录审计日志：

1. 注册与提交；
2. 删除与恢复；
3. 重建索引；
4. 批量下载；
5. 权限失败与异常访问。

## 八、容错与补偿设计

### 8.1 本地端

1. 单资源失败不阻断批量任务；
2. 每阶段失败记录错误码和错误信息；
3. 失败资源支持重新入队；
4. 网络失败后支持断点续传；
5. 缓存损坏时支持从日志恢复最近状态。

### 8.2 云端

1. 注册、提交、删除必须支持幂等；
2. 跨存储失败通过补偿任务修复；
3. 检索只读 `available` 数据，避免半成功数据泄露；
4. 索引更新失败时资源不得直接暴露给检索接口；
5. 定时对对象、元数据、向量三端做一致性巡检。

## 九、非功能需求与验收口径

### 9.1 目标指标

| 指标项 | 目标值 | 验证口径 |
|---|---|---|
| 单资源本地处理耗时 | 目标 P95 ≤ 4 秒 | 不含预览生成，统计描述 + 向量生成，稳定网络、主模型、无重试 |

| 检索召回效果 | 目标 Recall@10 达标 | 使用人工标注样本集评估 |
| 系统可用率 | 目标 ≥ 99.5% | 月度统计 |

### 9.2 检索质量评估

必须建立离线评估集，至少覆盖：

1. 图片；
2. 3D 模型；
3. 设计源文件；
4. 音频；
5. 字体/笔刷。

评估指标建议：

1. `Recall@5`
2. `Recall@10`
3. `MRR`
4. `nDCG`

## 十一、成本估算方法

### 11.1 成本拆分

总成本建议拆为：

1. 一次性生产成本：预览生成 + LLM 描述 + Embedding；
2. 固定存储成本：对象存储 + 元数据库 + 向量库存量；
3. 查询成本：检索请求、函数调用、向量检索；
4. 下载成本：下行流量与预签名访问；
5. 运维成本：日志、监控、告警。

### 11.2 测算口径

成本表必须显式标注：

1. 资源总量；
2. 平均文件大小；
3. 月检索次数；
4. 月下载流量；
5. 所选模型单价；
6. 是否使用冷存储或归档。

在未完成 PoC 测算前，所有成本数值仅作为示例区间，不作为正式预算承诺。

## 十二、风险与兜底方案

| 风险 | 触发条件 | 兜底方案 |
|---|---|---|
| GIF 识别不稳定 | 模型只读首帧或识别效果差 | 自动退化为关键帧拼图或静态首帧 + 辅助元数据 |
| LLM 输出波动大 | 格式失败率上升 | 固定参数、版本化 Prompt、切换备用模型 |
| 向量检索效果不稳定 | Recall@K 偏低 | 调整描述策略、Embedding 模型、过滤字段和索引参数 |
| 入库出现半成功数据 | 对象、元数据、向量任一写入失败 | 启用状态机与补偿任务，不暴露未 `available` 资源 |
| 上传中断 | 网络波动或限流 | 分片续传、指数退避重试 |
| 成本超出预期 | 调用量高于测算 | 缓存复用、限流、冷存储、模型分级策略 |
