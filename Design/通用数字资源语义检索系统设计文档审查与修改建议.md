# 通用数字资源语义检索系统技术设计文档审查与修改建议

## 1. 审查对象

- 文档名称：`通用数字资源语义检索系统（本地端+云端分离）技术设计文档（完善可下载版）`
- 审查目标：评估该设计文档在架构一致性、可实现性、接口与数据设计、性能与成本可信度、安全与运维完整性上的成熟度，并给出可直接用于修订文档的修改建议。

## 2. 总体结论

这份文档的优点是目标明确、阶段拆分清晰、业务闭环完整，且已经把“本地生产处理、云端检索服务、Agent 两步式调用”这条主链路表达出来了。作为项目愿景说明文档，它已经具备较强的沟通价值。

但如果将其定位为“最终定稿、可直接落地”的技术设计文档，目前仍存在几类关键缺口：

1. 若干核心假设尚未被验证，尤其是“动态图 GIF 作为 LLM 唯一输入”的兼容性与效果假设；
2. 全链路一致性、幂等性、状态机设计不够严谨，容易在真正实现时出现重复上传、脏数据、索引不同步等问题；
3. 若干指标、成本、周期表述偏乐观，缺少前提条件、基线数据和压测口径；
4. 本地端、云端、缓存层、Agent 层的接口边界还不够细，无法直接指导实现和测试。

建议将该文档从“最终定稿”调整为“评审修订稿”，完成本文列出的高优先级修订后，再作为实施基线版本。

## 3. 高优先级问题与修改建议

### 3.1 P0：将“动态图 GIF 作为唯一 LLM 输入”的假设补充为能力矩阵与降级方案

#### 问题

文档把“多视角/多状态资源合并为单张 GIF 预览图，并作为 LLM 唯一输入”设为核心前提，但没有说明不同多模态模型对 GIF 的真实支持情况，也没有定义当模型只读取首帧、自动抽帧失败、或根本不理解动画语义时的降级方案。

这不是文字细节问题，而是架构级风险。因为一旦 GIF 识别效果不稳定，3D 模型、多状态 UI 组件这类资源的描述质量会明显下降，后续 Embedding、检索和 Agent 调用都会被连锁影响。

#### 建议修改

建议在“预览图规范”和“LLM 描述生成模块”之间增加一节“模型输入兼容性策略”，明确以下内容：

1. 列出已选厂商/模型对 `png/webp/gif` 的实际支持情况；
2. 明确 GIF 的系统语义是“动画文件”还是“多帧信息载体”；
3. 增加统一降级策略：
   - 首选：直接传 GIF；
   - 不支持时：自动抽取关键帧并拼接为单张九宫格或横向联系图；
   - 再不支持时：退化为首帧静态图，并补充结构化元数据提示；
4. 将“唯一输入”从“仅 preview_file_path”修订为“标准化预览载体 + 资源类型 + 必要辅助元数据”。

#### 推荐修订表述

可改为：

“LLM 输入载体为标准化预览资源。静态资源优先使用 webp/png；动态或多状态资源优先使用 GIF。若目标模型对 GIF 支持不足，则由本地端自动生成关键帧拼图作为等价输入，并保留统一描述生成流程。”

### 3.2 P0：重新定义 `resource_id`、去重键和幂等键，避免身份语义混乱

#### 问题

文档中对 `resource_id` 的描述前后不够一致：

1. 术语定义中写的是“基于 `file_md5` 与临时 ID 生成”；
2. 数据结构中又写成“全局唯一 UUID”；
3. 上传部分写“同一 `resource_id` 重复上传直接覆盖更新”；
4. 去重逻辑又依赖 `resource_md5`。

目前实际上混合了三种不同语义：

- 资源内容指纹；
- 系统主键；
- 幂等控制键。

如果不拆开，后续会出现同一文件重传是否复用旧资源、重命名资源是否算新资源、同内容不同版本如何管理等实现争议。

#### 建议修改

建议把标识体系拆成四层：

1. `content_md5`：内容去重键，只反映原始文件内容；
2. `resource_id`：系统主键，建议 `UUIDv7` 或数据库生成，不承载业务语义；
3. `version_id`：可选，资源内容更新时新增版本；
4. `idempotency_key`：上传或入库请求的幂等控制键。

同时明确规则：

1. 同一 `content_md5` 可决定“跳过重新生成”还是“挂接到已有资源记录”；
2. `resource_id` 一经生成不可复用为别的内容；
3. 覆盖更新应定义为“更新版本”而不是“覆盖同一主键对应的全部数据”；
4. 云端接口幂等性应以 `idempotency_key` 或 `content_md5 + pipeline_version` 为准，而不是直接复用 `resource_id`。

### 3.3 P0：把“云端强一致”改为“显式状态机 + 最终一致 + 补偿机制”

#### 问题

文档同时涉及对象存储、关系型数据库、向量数据库三类异构存储，却使用了“事务机制：单资源操作同步强一致”这类表述。这个说法不够严谨，因为跨对象存储、关系库、向量库通常无法依靠单个事务实现严格强一致。

如果继续沿用现在的写法，开发阶段很容易误以为只要串行调用三个存储就算“强一致”，但一旦某一步失败，就会留下半成功数据。

#### 建议修改

建议把云端写入流程改写为“资源入库状态机”，至少定义以下状态：

- `registered`
- `object_uploaded`
- `metadata_saved`
- `vector_saved`
- `indexed`
- `available`
- `failed`
- `deleting`
- `deleted`

并补充以下机制：

1. 每一步写入都必须幂等；
2. 每一步完成后记录状态和错误码；
3. 失败时进入补偿流程，而不是笼统写“自动回滚”；
4. 检索服务仅对 `available` 状态资源开放；
5. 删除流程也走状态机，避免对象已删但向量仍可检索。

### 3.4 P0：补足上传接口协议，不要把“上传文件”和“写入索引”模糊成一个接口

#### 问题

文档当前只写了“POST 上传数据包、原始文件、预览图”，但没有说明：

1. 是同步接口还是异步任务；
2. 大文件分片是否走对象存储直传；
3. 元数据入库与向量写入由谁触发；
4. 客户端何时认为“上传完成”；
5. 失败重试的最小单元是什么。

这会导致客户端、云函数、对象存储、数据库之间边界不清，落地时很容易反复返工。

#### 建议修改

建议把上传协议拆成三段式：

1. `POST /resources/register`
   - 提交元数据摘要、`content_md5`、文件大小、资源类型；
   - 返回 `resource_id`、上传目标、是否已存在、分片策略。
2. 文件上传
   - 小文件走服务端代理上传；
   - 大文件走对象存储分片直传；
   - 客户端上报分片完成情况。
3. `POST /resources/commit`
   - 云端校验对象完整性；
   - 触发元数据落库、向量写入、索引构建；
   - 返回最终状态或异步任务 ID。

如果希望保留“单接口上传”描述，也应明确该模式只适用于小文件和 PoC 阶段。

## 4. 中高优先级问题与修改建议

### 4.1 P1：性能指标需要补充前提条件与测量口径

#### 问题

文档中的指标如“单资源本地处理 ≤4 秒”“1000 个资源 ≤30 分钟”“十万级检索 ≤2 秒”，目前更像目标值，而不是可直接承诺的工程指标。因为这些指标受以下因素强烈影响：

- 使用哪一家 LLM/Embedding 厂商；
- 网络带宽和时延；
- 是否启用并发；
- 预览图大小与 GIF 帧数；
- 向量维度与索引参数；
- 检索过滤条件是否命中分区。

#### 建议修改

建议把指标分成“目标值”和“验证口径”两栏，至少补充：

1. 测试数据规模；
2. 并发数；
3. 模型版本；
4. 网络环境；
5. 统计口径（P50、P95、最大值）；
6. 是否包含重试时间。

例如：

“十万级资源检索响应 ≤2 秒”应改为：

“在十万条、768 维向量、HNSW 默认参数、单次 Top20 检索、带资源类型过滤、无冷启动条件下，P95 响应时间目标为 ≤2 秒。”

### 4.2 P1：成本估算缺少计算公式，可信度不足

#### 问题

文档虽然给出了月度成本区间，但没有说明成本如何得出，尤其没有拆分：

- LLM 描述生成成本；
- Embedding 调用成本；
- 对象存储容量与下行流量；
- 数据库存储与请求数；
- 向量库容量、索引和 QPS 成本；
- Serverless 调用次数。

#### 建议修改

建议新增“成本模型说明”小节，按以下公式估算：

1. 单资源一次性生产成本；
2. 每月存储成本；
3. 每月检索请求成本；
4. Agent 下载流量成本；
5. 冷热分层带来的节省比例。

同时建议把“15-120 元”“200-600 元”改为“示例区间”，并注明测算前提。

### 4.3 P1：本地缓存结构需要区分“运行态缓存”和“对外标准数据结构”

#### 问题

当前统一数据结构体中既有本地绝对路径，又有云端 URL，又有向量数据，还试图作为“标准化数据包”传递给云端。这种设计会导致：

1. 本地运行字段和云端持久化字段混杂；
2. 绝对路径不具备跨机器可移植性；
3. 相同结构同时用于缓存、传输、接口返回时，字段演化成本很高。

#### 建议修改

建议拆成三层模型：

1. 本地处理实体：包含本地路径、处理状态、重试信息；
2. 云端注册/提交 DTO：仅包含云端所需字段；
3. 对外查询返回模型：仅返回 Agent 和业务方需要的字段。

同时将 `local_original_path`、`local_preview_path` 明确标记为“仅本地缓存字段，不参与云端接口透传”。

### 4.4 P1：检索质量评估缺失，无法判断系统是否真正“可用”

#### 问题

文档重点强调“向量检索准确率”，但没有定义如何评估准确率，也没有离线验证方案。若没有评估闭环，后续即使系统能跑通，也无法确定检索质量是否满足业务要求。

#### 建议修改

建议新增“检索效果评估方案”，至少包括：

1. 构建一批带人工标注的查询-结果样本集；
2. 指标采用 `Recall@K`、`MRR`、`nDCG`；
3. 针对不同资源类型分别评估；
4. 比较不同 Prompt、不同 Embedding 模型和不同分区策略；
5. 将“描述格式合格率”和“检索效果指标”分开管理，不要用前者替代后者。

### 4.5 P1：安全设计需要补足密钥托管和权限边界

#### 问题

当前安全章节有方向，但细节不够落地，尤其是：

1. Linux 端写“用 AES-256-GCM 加密”但未说明密钥放在哪里；
2. “全接口 JWT 鉴权”与“预签名 URL 无需额外鉴权”之间边界不清；
3. 上传、检索、下载、删除四类接口的权限粒度未定义；
4. 缺少审计日志与敏感操作追踪要求。

#### 建议修改

建议补充：

1. Linux 优先使用系统 Keyring，无法使用时再退化到本地主密钥方案；
2. 明确“业务接口走 JWT，预签名 URL 不走 JWT，只受对象存储签名与过期时间控制”；
3. 定义角色权限：上传者、检索者、管理员、Agent 服务；
4. 删除、批量导出、重建索引等高风险操作必须留审计日志。

## 5. 低到中优先级问题与修改建议

### 5.1 P2：文档用语需要从“宣传式”切换为“工程式”

#### 问题

文档中存在较多“最终定稿”“全程可直接落地”“彻底移除”“杜绝单点故障”“原生支持十万级、预留百万级扩展能力”这类结论式表达。对于方案介绍材料可以接受，但在实施设计文档里容易造成承诺过早。

#### 建议修改

建议把表述调整为：

- “当前方案基线”
- “默认实现路径”
- “目标能力”
- “建议约束”
- “待压测验证”

这样既保留方向性，也更符合工程文档风格。

### 5.2 P2：标题和版本信息存在重复与状态冲突

#### 问题

文档开头标题重复两次，且版本区写“最终定稿，可直接下载使用”，但正文又大量使用“待实现”“可后续扩展”等措辞，状态上并不完全一致。

#### 建议修改

建议：

1. 删除重复标题；
2. 将“文档状态”改为“V2.1 评审修订稿”或“实施基线稿”；
3. 在版本更新记录中写清本次修订目标和未决事项。

### 5.3 P2：部分资源类型定义仍偏模糊

#### 问题

例如 `material`、`other`、`design_file` 的边界还不够清晰，容易在实现时造成分类歧义，影响分区检索和统计。

#### 建议修改

建议新增“资源类型枚举说明表”，为每个类型补充：

1. 典型文件扩展名；
2. 归类规则；
3. 预览图生成策略；
4. 必填扩展元数据；
5. 检索过滤字段。

### 5.4 P2：Agent 工具协议还可再细化

#### 问题

Agent 章节说明了“两步式流程”，但还缺少错误码、分页、结果截断、阈值策略、空结果提示、下载权限失败场景等契约信息。

#### 建议修改

建议至少补充：

1. 检索接口返回分页参数与默认 TopK；
2. 结果字段中增加 `status`、`score`、`preview_available`；
3. 下载接口错误码约定，如资源不存在、地址生成失败、权限不足；
4. 当无匹配时返回推荐改写词或候选过滤条件。

## 6. 分章节修订建议

### 6.1 第一章：文档概述

建议新增“文档边界与不覆盖内容”，明确本文档是否覆盖：

- 资源审核与版权管理；
- 多租户与组织隔离；
- 控制台后台管理；
- 资源版本管理；
- 混合检索（关键词 + 向量）排序策略。

这样可以避免评审者把未覆盖内容误认为设计遗漏。

### 6.2 第二章：预览图生成模块

建议补充以下内容：

1. GIF 预览图的最大帧数、抽帧规则、背景色、渲染视角固定规则；
2. 设计源文件、音频、字体、其他类型的“已完成基础实现”是否已有真实依赖库，还是仅有占位方案；
3. 预览图失败时是否允许进入后续流程，还是直接标记资源不可检索；
4. 预览图质量校验规则，例如尺寸、黑屏、透明背景异常、文本截断等。

### 6.3 第三章：本地端业务模块

建议补充以下内容：

1. Provider 工厂的超时、熔断、厂商切换策略；
2. Prompt 版本号与模型版本的绑定方式；
3. 批量任务并发模型，是多进程、多线程还是异步 IO；
4. 缓存数据库的索引设计与清理策略；
5. 对“描述生成失败但预览图已完成”的中间态如何恢复。

### 6.4 第四章：云端服务层

建议增加一张“时序图或状态流转图”，至少说明：

1. 本地端注册资源；
2. 上传原始文件和预览图；
3. 云端写元数据；
4. 云端写向量；
5. 建索引；
6. 资源变为可检索状态；
7. 删除和回收站恢复路径。

如果没有这一页，后续开发和测试很难统一理解。

### 6.5 第五章：Agent 集成设计

建议明确 Agent 层不直接依赖内部存储结构，只消费稳定 API 契约；同时补充：

1. 是否支持按资源类型、尺寸、颜色、格式等结构化过滤；
2. 是否支持相似度阈值自动放宽；
3. 是否支持结果去重与多轮检索；
4. 下载链接默认有效期与最大可申请时长。

### 6.6 第六至第八章：安全、容错、非功能需求

建议把“安全要求”“容错要求”“性能要求”整理成一张验收清单，每项包含：

- 指标名称；
- 目标值；
- 验证方法；
- 责任模块；
- 是否上线门槛。

这样文档才能真正用于测试验收和里程碑评审。

### 6.7 第九至第十一章：实施计划、成本、风险

建议将计划和风险从“静态描述”升级为“可执行项目计划”：

1. 每阶段加入前置条件与退出条件；
2. 每阶段加入验收产物示例；
3. 风险表中补充“触发条件、监控指标、责任人、恢复时限”；
4. 成本章节加入“PoC 版 / 生产版”两套口径。

## 7. 推荐新增章节

建议在修订版文档中新增以下章节，以显著提升可实施性：

1. **能力验证章节**
   - 验证 GIF 输入兼容性；
   - 验证不同 LLM/Embedding 组合效果；
   - 验证 10 万级数据下的检索基线。
2. **接口协议章节**
   - 上传注册、提交、查询、删除的请求/响应示例；
   - 错误码与幂等规则；
   - 异步任务状态查询接口。
3. **状态机章节**
   - 本地处理状态机；
   - 云端入库状态机；
   - 删除与恢复状态机。
4. **测试与验收章节**
   - 单元测试；
   - 集成测试；
   - 检索效果评估；
   - 压测与故障演练。

## 8. 建议修订顺序

为了控制返工，建议按以下顺序修文：

1. 先修正核心假设和边界：GIF 输入、资源标识体系、云端一致性模型；
2. 再修正接口与数据结构：上传协议、缓存模型、查询返回结构；
3. 再补性能/成本/安全的验证口径；
4. 最后统一措辞、版本状态和章节组织方式。

## 9. 最终建议

这份文档已经具备“方案说明”和“实施方向确认”的价值，但还不建议直接作为开发唯一依据投入大规模实现。更稳妥的做法是：

1. 先完成一版架构修订；
2. 再补一版接口与状态机；
3. 最后基于 PoC 数据把性能、成本和效果指标固化。

完成以上修订后，这份文档会从“思路完整的方案稿”升级为“可以指导开发、测试、运维协作的实施设计稿”。
